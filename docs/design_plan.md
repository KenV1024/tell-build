## 1. ビジョンとゴール

**ビジョン**: 「作りたいものを一言伝えるだけで、要件定義→実装→テスト→公開→改善→報告まで完走する」ソフトウェア開発パイプライン。

**今回のゴール（/goal）**: パブリックベータユーザーを募集できる状態にする。
つまり「作者本人以外の人がforgeをクローンして、自分のClaude Codeで安全に使い始められる」こと。

**ゴールの受け入れ条件**:
- [ ] 初見の人がREADMEだけを見てセットアップ〜初回実行できる
- [ ] エンドツーエンド（E2E）で最低1プロジェクトがパイプラインを完走した実績がある
- [ ] 品質チェックリスト（`quality_checklist.md`）の必須項目が全て✅
- [ ] ベータ公開準備チェックリスト（`beta_readiness_checklist.md`）の必須項目が全て✅

## 2. 品質管理の基準（Anthropic社のエージェント構築原則に準拠）

Anthropicが公開しているエージェント設計のベストプラクティスを品質基準として採用する:

| 原則 | forgeでの実装 |
|---|---|
| **Human-in-the-loop**（人間の承認を挟む） | ゲート1（仕様承認）・ゲート2（デプロイ承認）の2箇所で必ずユーザー確認 |
| **単一責任・役割分離** | 1エージェント=1役割。越権禁止事項を各定義に明記 |
| **最小権限（Least Privilege）** | 各エージェントに必要最小限の`tools`のみ許可（例: qa-testerはEdit不可） |
| **予測可能なワークフロー** | 自由な相互呼び出しではなく、直列パイプライン＋状態ファイルで進行を固定 |
| **失敗の安全な扱い** | 差し戻し上限3往復→超過時はimprovement-analystが原因切り分け（無限ループ防止） |
| **観測可能性（Observability）** | 全フェーズが成果物ファイルを残す。pipeline_status.jsonでいつでも現在地を確認可能 |
| **段階的デプロイ** | 内部アルファ→限定ベータ→パブリックベータの3段階（下記マイルストーン） |

## 3. アーキテクチャ（確定済み・変更なし）

```
ユーザー依頼
   ↓
① spec-interviewer（要件ヒアリング・opus）
   ↓ ══ゲート1: 仕様承認══
② code-builder（実装・opus）
   ↓
③ qa-tester（動作確認・sonnet）
   ↓ ──FAILなら②へ（最大3往復）
   ↓ ══ゲート2: デプロイ承認══
④ release-manager（公開・sonnet）
   ↓
⑤ progress-reporter（報告・sonnet）→ ユーザー
   ↓
（公開後フィードバック）→ ⑥ improvement-analyst（opus）
        ├─ バグ → ②へ
        └─ 仕様変更 → ①へ（ゲート1やり直し）
```

- 正典: `shared/pipeline_rules.md`
- 司令塔: メインセッション（サブエージェントは他のサブエージェントを呼べないため）
- モデル方針: 難しい判断=Opus 4.8 / 定型作業=Sonnet 5、全員 `effort: high`

## 5. リスクと対策

| リスク | 影響 | 対策 |
|---|---|---|
| 生成コードの暴走（意図しないファイル操作） | 高 | 最小権限tools＋code-builderの作業範囲を`projects/<project>/`に限定 |
| ゲートのすり抜け（承認なしにpush/公開） | 高 | ゲート2をrelease-managerとメインセッションの両方で二重に強制 |
| 差し戻し無限ループ | 中 | 3往復上限＋improvement-analystエスカレーション |
| ベータユーザー環境での動作差異（OS・モデルアクセス権） | 中 | READMEに前提条件を明記（Claude Code版数・Opusアクセスの要否） |
| プロンプトインジェクション（外部コンテンツ経由の指示混入） | 中 | 品質チェックリストに検証項目として明記。外部データは指示ではなくデータとして扱う |
| Opus利用コスト（ベータユーザーの負担） | 低 | READMEにコスト目安とSonnetのみで動かすフォールバック設定を記載 |
